# Nexus Mail 操作流审计与测试缺口分析

针对用户核心操作流程的模拟分析，发现当前项目在“功能完备性”与“测试覆盖”上存在以下缺口：

## 1. 核心操作流模拟 (User Journey Simulation)

### 场景 A：多账户切换与数据隔离
- **模拟路径**: 用户从 `demo@nexus-mail.com` 切换到 `admin@nexus.local`。
- **预期**: 邮件列表立即清空并重新加载新账号的 Inbox。
- **缺口**: 目前 E2E 仅覆盖了单一账号的首次加载，**缺少多账号切换的断言**。

### 场景 B：文件夹深度导航
- **模拟路径**: 用户点击侧边栏的 "Sent" 文件夹。
- **预期**: `EmailList` 标题更新为 "Sent"，列表显示已发送邮件。
- **缺口**: **缺少文件夹切换测试**。需验证点击不同 Folder 后，`useMailbox` 是否正确触发了带 `folderId` 的 `invoke`。

### 场景 C：邮件详情读写与安全
- **模拟路径**: 收到包含 HTML 格式和恶意脚本的邮件。
- **预期**: HTML 正常渲染，但脚本被 `DOMPurify` 拦截。
- **缺口**: **缺少 XSS 安全测试**。应增加一个专门的集成测试验证注入攻击的拦截。

### 场景 D：附件下载 (阅读态)
- **模拟路径**: 用户点击邮件详情底部的附件列表进行下载。
- **预期**: 调用 `get_attachment` 并弹出保存对话框。
- **缺口**: **功能缺失**。目前 `EmailDetail` 组件尚未渲染附件列表，导致无法进行 E2E 验证。

---

## 2. 缺失功能 vs 缺失用例看板

| 类别 | 缺失点 | 建议操作 |
| :--- | :--- | :--- |
| **功能缺陷** | **搜索 UI (Search Bar)** | 后端 FTS 已就绪，前端需在 `EmailList` 增加搜索框。 |
| **功能缺陷** | **邮件操作 (Delete/Archive)** | 列表项或详情页缺少删除、存档、标记未读按钮。 |
| **功能缺陷** | **附件展示 (Read View)** | `EmailDetail` 需展示详情中的 `attachments` 数组。 |
| **测试缺口** | **分页/滚动加载** | 需测试滚动到底部时触发 `get_emails` 的 offset 增加。 |
| **测试缺口** | **同步错误反馈** | 模拟密码失效等网络异常，验证面包屑提示（Toasts）。 |
| **测试缺口** | **大邮件负载** | 测试包含 5MB+ 附件或超长 HTML 的渲染性能。 |

---

## 3. 下一步测试补全建议 (Priority)

1. **[E2E] 多维导航测试**: 覆盖“账号切换 -> 文件夹切换 -> 选中邮件”的一连串操作流。
2. **[Security] HTML 安全审计**: 注入恶意标签验证 `DOMPurify` 策略。
3. **[E2E] 异常路径模拟**: 拦截 `invoke` 调用并返回 `Error`，检查 UI 是否崩溃或有友好提示。
